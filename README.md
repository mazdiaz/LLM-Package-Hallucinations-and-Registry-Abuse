# LLM Package Hallucinations and Registry Abuse

Research investigating how often large language models recommend non-existent software packages, and the resulting supply-chain attack surface (slopsquatting).

## Problem

Code-generating LLMs frequently emit `import` statements referencing packages that do not exist in real registries. Attackers can register these hallucinated names with malicious code; users who follow model suggestions become victims. This phenomenon — "package hallucination" or "slopsquatting" — is an emerging software supply chain threat.

## Research Questions

- **RQ1:** How frequently do current PyPI and npm anti-abuse defenses (confusable-name rejection, quarantine, spam detection, squatting policy) catch hallucinated package names generated by state-of-the-art LLMs?
- **RQ2:** What features of a hallucinated package name predict evasion of registry defenses?
- **RQ3:** Can a lightweight classifier predict defense evasion with sufficient precision to serve as a pre-registration early-warning tool for registry operators?
- **RQ4:** How do hallucinated names distribute across the package-confusion taxonomy of Neupane et al. (2023)?

## Pivot

**Primary:** D — Registry Defense Benchmark. Test whether PyPI/npm anti-abuse defenses catch hallucinated names; build evasion classifier.

**Backup:** B — Agentic / MCP Hallucination (if D rejected).

## Methodology

1. Run N frontier LLMs on standard code-generation prompt datasets (reusing Spracklen et al. 2025 public pipeline where applicable).
2. Extract all package references (`import`, `require`).
3. Resolve each against live registry snapshots (PyPI, npm).
4. Classify registerability: claimed / unclaimed (READ-ONLY — no registration).
5. Apply each registry's published anti-abuse rules: predict blocked / passes.
6. Engineer features per hallucinated name (lexical, similarity, model consensus, popularity asymmetry).
7. Train classifier predicting defense evasion (methodology precedent: Andriushchenko 2026).
8. Cross-reference Neupane et al. (2023) package-confusion taxonomy.
